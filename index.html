Aegis is an internal AI platform for Chaayos — built so our engineers diagnose live systems in seconds instead of hours, so different teams stop rebuilding the same AI plumbing, and so our non-tech teammates can spin up their own contextual agents without writing code.

01 — The Problem

Three problems. We hit them every week.

Aegis exists because our current way of working has three specific, expensive gaps. Each one compounds the others.

PROBLEM 01

Debugging takes hours, not minutes.

Our engineers manually correlate logs, configs, code paths, and database state to root-cause production issues — across POS, ordering, cafe ops, and customer systems. There's no live, queryable understanding of how our services actually behave. Every investigation starts from zero.

HoursPer production investigation
PROBLEM 02

No centralized AI. Each team rebuilds the same thing.

Different Chaayos teams have started shipping their own bots, RAG stacks, and agent loops with no shared infrastructure. Today this doesn't exist as one platform — we'd end up with duplicated codebases, inconsistent UX, no shared governance, and AI that goes stale the moment a service changes.

N×Duplicate AI work across teams
PROBLEM 03

Non-tech teammates can't build their own agents.

Cafe ops, procurement, finance, and support teams hold most of our institutional knowledge. But creating a contextual AI today needs an engineer or an IT ticket. So that knowledge stays trapped in PDFs, spreadsheets, and inboxes — and the same questions get re-answered by hand every week.

0Self-service paths today

Why is product SKU-4471 not visible in the PO creation screen?

U
Procurement says SKU-4471 doesn't show up when creating a new PO. It exists in the catalog. What's happening?
→
code_rag · searchquery="product visibility filter PO creation" → matched ProductVisibilityService.filterForOrderContext() line 142
→
graph · execution_pathPOCreationController → ProductService.list() → ProductVisibilityService.filterForOrderContext() → checks Product.active && Product.visibility_scope contains user.region
→
db · readSELECT sku, active, visibility_scope FROM product WHERE sku='SKU-4471' → active=true, visibility_scope=['NORTH','EAST']
→
context · resolverequesting_user.region = 'WEST'
A
SKU-4471 is active, but its visibility_scope is set to NORTH and EAST only. The requesting user belongs to WEST, so filterForOrderContext() correctly excludes it.
Recommended actionUpdate visibility scope in catalog admin to include WEST, or confirm with merchandising whether the exclusion is intentional.

PO submission keeps getting rejected with code "VAL_017". Why?

U
User is trying to submit PO #82391 and keeps hitting "VAL_017". The error code isn't documented. What's failing?
→
code_rag · searchquery="VAL_017 validation" → POValidationRules.java:87 throws when totalAmount > vendorContract.creditLimit
→

→
db · readSELECT credit_limit FROM vendor_contract WHERE vendor_id='V-208' AND active=true → ₹4,00,000
A
PO #82391 totals ₹4,82,000, exceeding vendor V-208's active credit limit of ₹4,00,000. Validation is working as designed — VAL_017 in POValidationRules.java:87.
Recommended actionSplit the PO, request a credit-limit revision via vendor management, or split the order across vendors. No code defect.

Inventory prices don't match what PO service publishes. Why the drift?

U
PO says ₹212 for SKU-3120 but inventory shows ₹187 after sync. What's happening across the boundary?
→
graph · service_dependencyPO_Service.publishPriceEvent() → kafka.topic("price-updates") → Inventory_Service.PriceConsumer → InventoryPriceMapper.fromEvent()
→
code_rag · fetchInventoryPriceMapper.fromEvent() applies: net_price = event.price * (1 - vendor_default_discount)
→
db · readSELECT vendor_default_discount FROM inventory_vendor_config WHERE vendor='V-310' → 0.1179
A
PO publishes gross price (₹212). Inventory mapper applies an 11.79% vendor-default discount, producing ₹187. The systems represent different price stages — the contract between them isn't aligned.
Recommended actionStandardize the published price as net-of-discount, or remove the redundant discount step. Schedule contract review with both service owners.

PO approvals stopped going out from staging. No deploy. What changed?

U
Approval emails stopped on staging this morning. We didn't deploy. Where do I look?
→
code_rag · searchquery="approval email send" → ApprovalNotifier.send() reads notification.enabled flag before dispatch
→
db · readSELECT key, value, updated_at, updated_by FROM config_registry WHERE service='po-approval' AND env='staging' AND key='notification.enabled' → value=false, updated_at=2026-05-24 02:14, updated_by=ops-bot
A
The notification.enabled flag for po-approval on staging was flipped to false at 02:14 by ops-bot. ApprovalNotifier.send() short-circuits when it's off.
Recommended actionVerify whether the change was intentional (scheduled maintenance?). If not, restore the flag via Aegis Console's config registry. Full audit trail available.
04 — Architecture

Four layers. Top-down governance.

Every request flows through governance before it reaches the AI runtime. The AI runtime only touches data the policy explicitly allows. Safe to roll out across our production systems.

L1
InteractionAegis Pulse

Framework-agnostic embedded UI that drops into any Chaayos app. Captures user input and injects runtime context (service, user, screen, environment) before forwarding to Aegis Core. Zero changes to the host app's business logic.

L2
GovernanceAegis Core

Central nervous system. Java / Spring Boot. Configuration registry, hierarchical policy resolution, intent routing, tenant isolation, and context resolution per request.

L3
AI RuntimeAegis Cortex

Centralized execution engine. Hosts all specialized agents (Diagnostic, Analysis, SQL, Support, Custom RAG). Manages tools, runs retrieval-augmented reasoning, abstracts model providers (OpenAI, Anthropic).

L4
IntelligenceLive retrieval

Code RAG (Spoon / JavaParser + Neo4j graph + pgvector embeddings), read-only database verification, log and trace retrieval (Phase 2+). All sources are governed and audit-logged.

05 — Governance & Metrics

Read-only. Multi-tenant. Fully auditable.

Built to roll out safely across Chaayos production systems. Every initial phase is read-only. Every action is logged. Every agent sees only what policy permits.

Multi-tenant isolation

Strict separation by service, environment, user, and data source. No data leaks between Chaayos teams, services, or roles.

Read-only by design

No write, update, or delete against any Chaayos system. Autonomous remediation is explicitly out of scope for the initial phases.

Policy-based tools

Agents access only the tools and data sources permitted by service, role, and environment policy.

Full audit trail

Every query, retrieval, tool call, raw prompt, and model response is logged. Compliance and forensics ready.
